## Длинные строки и переносы (Python)

- Максимальная длина строки — 79 символов (согласно PEP 8)
- Избегай обратного слэша `\` для переноса строк, кроме крайних случаев

- Для длинных вызовов функций используй круглые скобки с переносом аргументов:

```python
result = some_function(
    first_arg,
    second_arg,
    third_arg_with_long_name,
)
```

- Для длинных логических выражений переноси каждое условие на отдельную строку:

```python
is_valid = (
    condition_one
    and condition_two
    and another_long_condition
)
```

- Для длинных текстовых строк применяй неявное объединение в скобках:

```python
message = (
    "Это длинное сообщение, которое "
    "разбито на несколько строк, "
    "но в рантайме остаётся одной строкой."
)
```

- При использовании форматтера (например, Black) следуй его стилю без ручной коррекции

## FastAPI (веб-API)

- Используй FastAPI для построения HTTP API, опираясь на официальную документацию и типизацию. [web:120][web:123]
- Базовая структура:
  - создаём `FastAPI()` приложение;
  - описываем path operations (`@app.get`, `@app.post` и т.п.) с аннотированными параметрами и схемами;
  - запускаем через Uvicorn/Gunicorn. [web:120][web:117]
- Для входных/выходных данных используй Pydantic‑модели (v1/v2 по проекту), а не «сырые» dict; схемы описывают контракт API. [web:120][web:123]
- Активно используй зависимости (`Depends`) для:
  - подключения БД/сессий/клиентов;
  - авторизации/аутентификации;
  - общих кусков бизнес‑логики/валидации, которые переиспользуются в нескольких эндпоинтах. [web:125][web:116][web:119]
- Разделяй слои:
  - роутеры (`APIRouter`) только принимают/отдают данные и вызывают доменные/сервисные слои;
  - доменная логика не зависит от FastAPI и веб‑слоя;
  - доступ к БД, кэшу, внешним сервисам — через репозитории/адаптеры. [web:123][web:116]
- Стремись к явным схемам ошибок и ответов (статусы, модели ошибок), не возвращай произвольные структуры без описания в OpenAPI. [web:120][web:123]


## Aiogram (Telegram-боты)

- Используй актуальную версию aiogram 3.x и опирайся на официальную документацию. [web:102][web:112][web:121]
- Базовая архитектура:
  - создаём `Bot` с токеном;
  - создаём `Dispatcher` как корневой `Router`;
  - подключаем отдельные `Router` по фичам/модулям;
  - запускаем бота через `dp.start_polling(bot)`. [web:112][web:113][web:124]
- Хендлеры — это `async`‑функции с фильтрами (`CommandStart`, `F.text`, кастомные фильтры); избегай «гигантских» хендлеров, разумно дели по фичам. [web:112][web:98]
- Для сложных сценариев используй FSM/Scenes (StatesGroup/Scene), не храни состояние в глобальных переменных. [web:114][web:115]
- Разделяй слои:
  - хендлеры только принимают апдейт и дергают доменную логику;
  - бизнес‑логика и работа с БД/сервисами живут в отдельных сервисах/модулях, без прямой завязки на aiogram API. [web:112][web:124]
- Локализацию, тексты сообщений и шаблоны выноси в отдельные ресурсы (например, FTL/шаблоны), не хардкодь большие тексты в хендлерах.

## Локализация и FTL (Fluent)

- Если в проекте уже используются `.ftl` файлы (Fluent), все пользовательские сообщения, тексты UI и ответы пользователю следует выносить в FTL, а не хардкодить строки в коде. [web:82][web:88][web:95]
- При добавлении нового текста для пользователя:
  - создай/используй соответствующий ключ в `.ftl` файле;
  - в коде ссылайся на ключ локализации, передавая параметры через Fluent API. [web:82][web:85][web:88]
- Избегай конкатенации локализуемых строк в коде; формируй сообщения через FTL с параметрами и плейсхолдерами. [web:82][web:92][web:95]
- Не смешивай в одном сообщении текст и «сырой» HTML/разметку без необходимости; выноси всё, что можно, в FTL, используя возможности Fluent (атрибуты, варианты, селекторы). [web:82][web:84][web:95]
- При изменении формулировок сообщений:
  - по возможности изменяй текст в `.ftl`, не меняя идентификатор;
  - если меняется смысл/контракт сообщения, заводи новый ключ, а старый помечай как deprecated. [web:87][web:92][web:95]
- Для многоязычных проектов не добавляй новую логику, которая жёстко завязана на конкретный язык (например, ручные склонения/плюрализацию) — используй возможности Fluent (selectors, pluralization). [web:82][web:93][web:95]

## Архитектура и принципы (DRY, SOLID, Clean/DDD/FSD/Onion)

- Старайся соблюдать принципы DRY: не дублируй код; при повторении логики выделяй общие части в функции, методы или отдельные модули, но без over-engineering.
- Соблюдай SOLID там, где это оправдано: не ломай существующие принципы ради «красивого» кода, если это усложняет решение без пользы.

- Отдавай приоритет чистой архитектуре: разделяй доменный слой, приложение/сервисы, инфраструктуру.
- Не смешивай доменную логику с инфраструктурой (БД, HTTP, брокеры и т.п.); доменные сущности не должны знать о внешних деталях.

- При проектировании модулей и слоёв опирайся на подходы Clean Architecture / DDD / Onion:
  - зависимости направлены из внешних слоёв во внутренние;
  - доменная модель и use-cases не зависят от фреймворков и конкретных библиотек;
  - инфраструктурные детали оформляй адаптерами/реализациями интерфейсов.

- Для фронтовых/модульных структур по типу FSD:
  - группируй код по фичам/домейн‑областям, а не только по техническому типу (controllers, services и т.п.);
  - выноси общие компоненты/модули в отдельные слои shared/core только если реально есть переиспользование, а не «на всякий случай».

- Не навязывай сложные архитектурные паттерны простым скриптам или утилитам: сначала простое решение, затем постепенное усложнение по мере роста проекта.
- При внесении архитектурных изменений учитывай существующий стиль проекта: не смешивай несколько архитектурных подходов без необходимости.
- Стремись к тестопригодности: доменная логика должна быть легко покрываема модульными тестами без поднятия инфраструктуры.

## Валидация данных (Pydantic)

- Для входных/выходных DTO в API, background jobs и интеграциях используй Pydantic-модели вместо «голых» dict/tuple. [web:45][web:46][web:51]
- Все поля в моделях должны иметь явные type hints; по возможности используй специализированные типы (EmailStr, HttpUrl, constr, conint и т.п.). [web:44][web:47][web:48]
- Простую бизнес-логику валидации допускается описывать валидаторами (`@field_validator`, `@model_validator`), но не превращай модель в «божественный объект» — сложные правила выноси в отдельные сервисы/функции. [web:47][web:48]
- Не дублируй валидацию: если Pydantic уже проверяет тип/диапазон, не добавляй поверх ручные проверки без необходимости. [web:44][web:48][web:56]
- При работе с performance‑критичным кодом не включай лишнюю валидацию (используй `model_validate`, `model_validate_json`, `model_construct`, `validate_assignment` по необходимости). [web:44][web:48][web:53]
- Для взаимодействия с внешним миром (HTTP, очереди, файлы) отдавай/принимай данные через Pydantic-модели, а не «сырые» dict, где это возможно. [web:45][web:46][web:51]

## Форматирование и стиль

- Отдавай предпочтение явным именованным параметрам вместо длинных списков позиционных.
- Если функция начинает «раздуваться», предлагай разбиение на более мелкие, но только если это явно обсуждается в задаче.

- Группируй импорты: стандартная библиотека, сторонние, локальные; между группами пустая строка. [web:23][web:26]
